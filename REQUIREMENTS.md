# BusPool — 公交日票共享平台 · 需求文档

## 一、项目背景

Durham 是英国一座大学城，学生日常出行依赖公交。一张 Day Ticket（Dayrider）售价 £1.6，允许当天无限次乘坐。但很多学生只用了几个小时就不再乘车，剩余价值被浪费。

**BusPool** 让这些闲置的 Day Ticket 流通起来：用完上午的学生将条码照片上传到平台，下午需要乘车的学生可以借用，实现"一人购票，全天复用"。

## 三、用户角色

| 角色 | 描述 |
|---|---|
| **上传者 (Seller)** | 买了 Day Ticket 但用不了一整天的学生，上传条码照片获取积分 |
| **借用者 (Borrower)** | 需要乘车的学生，用积分借用别人的条码 |
| **管理员 (Admin)** | 平台运营者，处理申诉、审核异常 |

## 四、核心页面与工作流

### 页面 1：首页（票务广场）

**功能**：展示当前可用的 Day Ticket 列表

**展示内容**：
- 每张票显示：上传者头像、上传时间、信誉分、历史成功率、所需积分
- 实时更新：有新票上传时无需刷新即可看到
- 筛选：按上传时间、信誉分排序
- 空状态：当没有可用票时，显示"暂无可用票，稍后再来"

**用户行为**：
- 浏览票列表 → 点击某张票 → 进入票详情

---

### 页面 2：票详情

**功能**：查看单张票的完整信息，决定是否借用

**展示内容**：
- 条码照片（模糊预览，借用后才显示清晰版）
- 上传者信息：头像、昵称、信誉分（0-100）、累计上传次数、历史成功率（如 95% 有效）
- 票信息：类型（Dayrider）、购买时间、有效区域
- 借用所需积分
- 倒计时：此票自动过期时间（当天 23:59）

**用户行为**：
- 点击"借用此票" → 确认弹窗 → 积分冻结 → 条码照片清晰展示 → 上车扫码

---

### 页面 3：上传票（发布闲置票）

**功能**：上传自己用不完的 Day Ticket

**流程**：
1. 拍摄或选择条码照片
2. 选择票类型（Dayrider / DaySaver）
3. 填写购买时间
4. 声明"本人今天已使用过此票"（勾选确认）
5. 提交上传

**限制**：
- 上传者信誉分必须 ≥ 30
- 同一票条码不能重复上传
- 每人每天最多上传 3 张票

**上传后**：
- 状态变为"待借用"
- 其他用户可在首页看到
- 被借用并确认有效后，上传者获得积分

---

### 页面 4：我的借用（借用记录）

**功能**：查看自己借过的票和当前正在使用的票

**状态流转**：
```
借用中 → 等待确认 → 已完成
         ↓
       已失效（票不能用）
```

**展示内容**：
- 正在使用中的票：条码照片（清晰）、借用时间、剩余有效时间
- 历史借用记录：票状态、是否有效、花费积分

**用户行为**：
- 查看当前票的条码 → 上车扫码 → 使用后点击"确认结果"

---

### 页面 5：确认结果（关键流程）

**功能**：借用者上车扫码后，确认票是否有效

**两种确认按钮**：
- ✅ **能用** — 票成功通过扫码器，正常乘车
- ❌ **不能用** — 票被扫码器拒绝，无法乘车

**确认"能用"后**：
- 冻结的积分转给上传者
- 上传者信誉 +2
- 交易完成

**确认"不能用"后**：
- 冻结的积分退回借用者
- 上传者信誉 -10
- 票标记为"已失效"
- 借用者可提交申诉（附照片等证据）

**超时规则**：
- 借用后 **4 小时**内未确认 → 系统自动按"能用"结算
- 借用后 **1 小时**内未确认 → 发送提醒通知
- 超时时间设为 4 小时而非 24 小时，原因：Day Ticket 当天有效，4 小时足够覆盖单次出行场景，避免长时间占用资源

---

### 页面 6：积分钱包

**功能**：查看积分余额、收支明细

**展示内容**：
- 当前积分余额
- 积分明细列表：
  - 收入：上传的票被使用 +XX 积分
  - 支出：借用票 -XX 积分
  - 申诉退款 +XX 积分
- 积分兑换（未来可接入真实货币）

**积分定价参考**：
- 上传一张票被使用：+5 积分
- 借用一张票：-5 积分
- 新用户注册赠送：10 积分

---

### 页面 7：个人中心

**功能**：用户信息、信誉管理

**展示内容**：
- 头像、昵称、注册时间
- 信誉分（0-100）及等级标签
  - 80+ 优秀（绿色）
  - 50-79 良好（蓝色）
  - 30-49 一般（橙色）
  - 30 以下 限制（红色，禁止上传）
- 统计数据：上传次数、借用次数、成功率
- 信誉规则说明

**用户行为**：
- 编辑个人信息
- 查看信誉详情
- 查看申诉记录

---

### 页面 8：申诉中心

**功能**：当借用者确认"不能用"时，上传者可发起申诉

**流程**：
1. 上传者收到"票被确认无效"通知
2. 上传者可提交申诉：说明情况 + 证据（如自己当天使用照片）
3. 管理员审核
4. 裁定结果：维持原判 / 恢复信誉 / 恢复积分

---

### 页面 9：用户协议（首次使用必读）

**功能**：法律免责 + 使用规则说明

**核心条款**：

**1. 平台定位**
> BusPool 仅提供信息共享服务，撮合用户之间的票务资源共享。平台本身不售票、不验票、不保证票务的有效性。所有交易行为发生在用户之间，平台不作为交易方。

**2. 免责声明**
> - 平台不对上传票务的真实性、有效性作任何担保
> - 用户使用他人票务时，如被公交公司拒绝或处罚，责任由用户自行承担
> - 平台不承担因票务无效导致的任何损失（包括但不限于无法乘车、被罚款等）
> - 用户理解并同意：共享他人 Day Ticket 可能违反公交公司条款，由此产生的一切后果由用户自行承担

**3. 用户责任**
> - 上传者保证所上传票务为本人当日合法购买且真实有效
> - 借用者理解票务来自第三方，存在无法使用的风险
> - 用户不得利用平台进行欺诈、刷分等违规行为
> - 条码照片在借用期间可见，借用者不得截图转发给第三方（技术上无法完全防止，但属于违规行为，一经发现封号处理）

**4. 积分规则**
> - 积分仅限平台内使用，不可兑换现金
> - 平台有权调整积分规则，提前通知用户
> - 异常积分交易（如刷分）将被冻结并审核

**5. 账号规则**
> - 每人限一个账号
> - 信誉分低于 30 将限制上传功能
> - 严重违规（如多次上传假票）将永久封号

**触发时机**：
- 用户首次注册时，必须勾选"已阅读并同意用户协议"才能完成注册
- 上传票务时，再次确认"本人保证此票真实有效"
- 借用票务时，再次确认"我理解此票来自第三方，存在无法使用的风险"

## 六、安全与防作弊机制

| 风险 | 应对 |
|---|---|
| 上传假票/旧票 | 信誉系统：假票导致信誉 -10，低于 30 禁止上传 |
| 借用者白嫖（说不能用但其实用了） | 信誉系统 + 4h 超时自动结算（Day Ticket 当天有效，4h 足够覆盖单次出行） |
| 同一票同时借给多人 | 票被借用后立即锁定，状态改为"使用中" |
| 条码照片被截图转发 | 技术手段：借用期间才显示清晰版，过期后模糊处理。法律手段：用户协议明确禁止截图转发，违规封号。技术上无法 100% 防止，但通过协议明确责任归属，降低平台法律风险 |
| 刷积分 | 同一票每天限制上传次数，异常交易人工审核 |

## 九、冷启动策略（MVP 必做）

**问题**：平台上线初期没有用户上传票务，新用户看到空广场直接离开。

**解决方案**：

1. **运营者自给初始供给**
   - MVP 上线前，运营者（你本人）购买 5-10 张 Day Ticket
   - 每天上传 2-3 张到平台，确保广场始终有票可借
   - 持续 2 周，直到有自然用户开始上传

2. **种子用户激励**
   - 邀请 10-20 个同学作为种子用户
   - 种子用户注册送 20 积分（普通用户 10 积分）
   - 种子用户首次上传成功额外奖励 10 积分

3. **时间窗口运营**
   - 早上 8-9 点：运营者上传当天第一批票（覆盖上午出行）
   - 下午 2-3 点：鼓励有票的用户上传（覆盖下午出行）
   - 晚上 6-7 点：鼓励有票的用户上传（覆盖晚间出行）

4. **社群引导**
   - 在 Durham 大学微信群/QQ 群推广
   - 发布使用教程和省钱对比（单程 £1.5 vs 借用 £0.3）
   - 收集早期反馈，快速迭代

## 十、管理后台（Admin Dashboard）

**入口**：`/admin`，仅管理员可访问（Supabase Auth 角色控制）

**核心原则**：管理后台是平台运营者的"驾驶舱"，用于处理系统无法自动处理的异常情况、监控平台健康度、以及手动干预紧急事件。

---

### 10.1 数据看板（Dashboard 首页）

**功能**：一目了然的平台运营数据

**展示内容**：
- 今日关键指标
  - 活跃用户数 (DAU)
  - 今日上传票数 / 借用次数
  - 票有效率（确认有效 / 总确认）
  - 平均借用等待时间
- 趋势图表（近 7 天 / 30 天）
  - 用户增长曲线
  - 每日交易量
  - 票有效率变化
  - 积分流通总量
- 实时状态
  - 当前可用票数
  - 正在使用中的票数
  - 待审核的申诉数
  - 待处理的异常告警

---

### 10.2 用户管理

**功能**：查看和管理所有用户

**用户列表**：
- 搜索：按昵称、邮箱、ID 搜索
- 筛选：按信誉分、注册时间、账号状态
- 排序：按注册时间、信誉分、上传次数

**用户详情**：
- 基本信息：头像、昵称、邮箱、注册时间
- 账号状态：正常 / 限制 / 封禁
- 积分余额、信誉分
- 统计：上传次数、借用次数、成功率
- 操作历史：所有交易、申诉记录

**管理员操作**：
- 调整积分（手动发放 / 扣除）
  - 输入：积分数量 + 原因（必填）
  - 正数为发放，负数为扣除
  - 操作记录写入日志
- 调整信誉分
  - 输入：信誉分变化 + 原因（必填）
- 修改账号状态
  - 正常 → 限制（禁止上传）
  - 正常 → 封禁（禁止所有操作）
  - 封禁 → 正常（解封）
- 查看该用户的所有操作日志

---

### 10.3 票务管理

**功能**：查看和管理所有票务

**票务列表**：
- 搜索：按票 ID、上传者搜索
- 筛选：按状态（可用 / 使用中 / 已完成 / 已过期 / 已失效）、票类型、上传时间
- 排序：按上传时间、借用次数

**票务详情**：
- 条码照片
- 上传者信息
- 票类型、购买时间、过期时间
- 状态、被借用次数
- 所有交易记录

**管理员操作**：
- 强制下架票务（标记为无效）
  - 输入：下架原因（必填）
- 查看条码原图（调试用）
- 查看该票的所有借用记录

---

### 10.4 交易管理

**功能**：查看和管理所有交易记录

**交易列表**：
- 搜索：按交易 ID、用户 ID、票 ID 搜索
- 筛选：按状态（待确认 / 已确认有效 / 已确认无效 / 已超时）、时间范围
- 排序：按时间、积分金额

**交易详情**：
- 交易双方信息（上传者 / 借用者）
- 票务信息
- 积分变化
- 确认时间 / 超时时间
- 申诉记录（如有）

**管理员操作**：
- 手动确认交易状态
  - 将"待确认"交易标记为"有效"或"无效"
  - 输入：确认结果 + 原因（必填）
- 撤销交易
  - 回滚积分变化
  - 输入：撤销原因（必填）

---

### 10.5 申诉审核

**功能**：处理用户提交的申诉

**申诉列表**：
- 状态筛选：待审核 / 已裁定 / 已驳回
- 按时间排序

**审核流程**：
1. 查看申诉详情：申诉人、原因、证据照片
2. 查看关联交易：原始票务、借用记录
3. 做出裁定：
   - 维持原判（借用者说不能用 → 确实不能用）
   - 恢复上传者信誉和积分（借用者误报或恶意举报）
   - 部分处理（酌情调整）
4. 输入：裁定说明（必填，通知给申诉人）

---

### 10.6 积分手动发放（应急功能）

**功能**：手动给指定用户发放或扣除积分

**使用场景**：
- 系统 Bug 导致积分计算错误，需要手动修正
- 用户反馈积分未到账，核实后手动补发
- 活动奖励发放（如种子用户奖励）
- 惩罚扣除（如确认的欺诈行为）

**操作流程**：
1. 搜索用户（邮箱 / ID / 昵称）
2. 选择操作类型：发放 / 扣除
3. 输入积分数量
4. 输入原因（必填，至少 10 个字符）
5. 确认弹窗：显示操作前后积分余额
6. 提交 → 积分立即到账 → 写入操作日志

**安全限制**：
- 单次操作上限 1000 积分
- 每日操作上限 5000 积分
- 所有操作不可撤销（通过日志追溯）
- 操作记录永久保存

---

### 10.7 操作日志（审计追踪）

**功能**：记录管理员的所有操作，确保可追溯

**记录内容**：
- 操作时间
- 操作管理员
- 操作类型（调整积分 / 调整信誉 / 封禁用户 / 下架票务 / 审核申诉 / 手动发放）
- 操作对象（用户 ID / 票 ID / 交易 ID）
- 操作详情（变化前 → 变化后）
- 操作原因

**日志查询**：
- 按管理员筛选
- 按操作类型筛选
- 按时间范围筛选
- 导出 CSV

## 十一、页面导航结构
